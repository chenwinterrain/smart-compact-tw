# 策略性對話管理指引

在長對話中維持回應品質，透過階段性摘要和對話切換策略管理上下文。

---

## 一、核心原則

透過回顧對話輪次和資訊量，在品質下降之前主動提醒使用者。
任一觸發條件成立即提醒，不需要全部滿足。

---

## 二、主動提醒的觸發條件

**以下規則在整個對話中持續生效，不需使用者觸發。**
**任一條件成立即觸發提醒（OR 邏輯）。**

### 條件 A：對話輪次門檻

回顧目前對話的來回輪次和每輪的資訊量，依對話類型判斷：

| 對話類型 | 特徵 | 開始留意的輪次 |
|---------|------|-------------|
| 輕量對話 | 短問短答、討論、閒聊 | 40 輪 |
| 中量對話 | 有分析、有產出文字內容 | 30 輪 |
| 重量對話 | 上傳檔案、產出書狀、讀取判決、產出程式碼 | 20 輪 |

**判定方式**：回顧對話內容，判斷屬於哪種類型。如果對話中混合了多種類型，
以最重的類型為準。

### 條件 B：上下文緊縮感知

不論輪次多少，當 Claude 感知到上下文空間明顯緊縮時，直接觸發。

### 觸發時的提醒方式

```
⚡ 這個對話已經累積不少內容了，繼續下去可能會開始遺失前面的細節。
要不要我整理一份接續摘要，你帶到新對話繼續？
```

### 不觸發的情況

- 使用者正在等待一份重要輸出（書狀、分析報告等）時，先完成產出再提醒
- 輪次和資訊量都還在安全範圍內，且上下文空間充足

---

## 三、接續摘要與交接協定（Handoff Protocol）

當使用者同意做階段性摘要時，依序執行：
**確認清單 → 產出摘要 → 提供啟動指令**。

### 3-1 產出前確認清單

逐項自我檢查，確保關鍵資訊不遺漏：

- [ ] **關鍵結論**：目前已達成的分析結論或決策是否都已記錄？
- [ ] **檔案清單**：對話中引用或上傳過的檔案是否都已列出？每個檔案標明用途和必要性。
- [ ] **進度狀態**：目前在整體工作流程中的位置是否清楚？
- [ ] **待辦事項**：接下來要做的步驟是否明確？
- [ ] **使用者偏好**：對話中使用者表達過的偏好或限制是否已記錄？（例如「這個案子不要引用 XX 判決」）
- [ ] **Skill 指定**：下一步需要觸發哪個 Skill？是否已在摘要中明確標示？
- [ ] **工作地圖**：如果是多產出物的案件/專案，是否已建立樹狀結構的工作地圖？每個產出物的狀態是否標記正確？

如果有任何項目無法確認，先詢問使用者再產出摘要。

### 3-2 摘要格式與輸出方式

摘要必須以 `<!-- HANDOFF-START -->` 開頭、`<!-- HANDOFF-END -->` 結尾。
新對話的 Claude 看到此標記時，應自動進入接續模式（見 3-4）。

**輸出方式（二擇一）：**

| 模式 | 說明 | 適用情境 |
|------|------|---------|
| **對話內文字**（預設） | 直接在對話中輸出 Markdown 文字，使用者複製貼上即可 | 一般交接、摘要較短 |
| **檔案輸出** | 產出 `.md` 檔案供下載保存 | 摘要很長、需保存多份交接紀錄、大型案件多階段交接 |

**觸發規則：**
- 預設使用對話內文字模式。
- 當使用者說「存檔」「輸出檔案」「下載摘要」時，改用檔案輸出模式。
- 當摘要內容明顯過長（例如大型案件、多份書狀交接），Claude 應主動建議改用檔案輸出。

```markdown
<!-- HANDOFF-START -->
# 接續摘要

## 案件/工作項目
[案件名稱或工作項目描述]

## 已完成
- [完成事項 1]
- [完成事項 2]

## 關鍵結論
- [重要結論 1]
- [重要結論 2]

## 進行中
- [目前正在做的事]

## 下一步
1. [接下來第一步]
2. [接下來第二步]

## 需要上傳的檔案
| # | 檔案名稱 | 用途 | 必要性 |
|---|---------|------|-------|
| 1 | [檔名] | [這份檔案在後續工作中的用途] | 🔴 必要 / 🟡 建議 / ⚪ 參考 |
| 2 | [檔名] | [用途] | [必要性] |

## 工作地圖
[用樹狀結構呈現整個案件/專案的全局進度，包含所有產出物及其狀態]
```
案件名稱/
├── 📄 民事起訴狀.docx ── ✅ 已完成（對話 2）
├── 📄 證據清單.docx ── ✅ 已完成（對話 2）
├── 📄 刑事附帶民事起訴狀.docx ── 🔨 進行中
├── 📄 聲請假扣押狀.docx ── ⏳ 待處理
└── 📋 案情摘要.md ── ✅ 已完成（對話 1）
```
[上方為範例，實際依案件內容產出。狀態符號：✅ 已完成、🔨 進行中、⏳ 待處理]

## 指定 Skill
下一步應使用：`[skill-name]`（[簡述原因]）

## 注意事項
- [需要特別記住的限制或偏好]

## 啟動指令
> [預擬的第一句話，使用者可直接複製貼上或修改後使用]
<!-- HANDOFF-END -->
```

### 3-3 啟動指令的撰寫規則

摘要末尾的「啟動指令」是讓使用者在新對話中**接在摘要後面貼上**的第一句話。
撰寫原則：

1. **具體**：直接說明要做什麼，不要寫「請繼續」這種空泛指令。
2. **銜接下一步**：指令內容對應摘要中「下一步」的第一項。
3. **包含 Skill 觸發詞**：如果下一步需要特定 Skill，指令中要包含該 Skill 的觸發關鍵字。

**範例：**
- ❌ `請根據上方摘要繼續工作。`
- ✅ `請根據上方摘要，幫我寫民事起訴狀。原告資訊和案情事實都在摘要裡了。`
- ✅ `接續上方摘要，繼續審查書狀。我會重新上傳 docx 檔。`
- ✅ `接續上方摘要，幫我篩選判決。我的爭點和篩選條件在摘要的「關鍵結論」裡。`

### 3-4 新對話接收端行為（Claude 端規則）

當 Claude 在對話的**第一則訊息**中偵測到 `<!-- HANDOFF-START -->` 標記時：

1. **確認接收**：回覆「已讀取接續摘要」並簡要複述案件/工作項目名稱和下一步。
2. **主動找檔**：比對摘要中「需要上傳的檔案」表格，檢查使用者是否已在本次對話中上傳對應檔案。
   - 已上傳（檔案出現在對話附件中）→ 主動讀取內容，告知使用者已載入。
   - 🔴 必要檔案未上傳 → 列出缺少的檔案名稱和用途，請使用者補上傳後再開始。
   - 🟡 建議 / ⚪ 參考檔案未上傳 → 告知但不阻擋工作進行。
3. **顯示工作地圖**：如果摘要中有「工作地圖」區塊，將其呈現給使用者，確認目前要處理的是哪一項。
4. **載入 Skill**：依摘要中「指定 Skill」欄位觸發對應的 Skill。
5. **套用注意事項**：將「注意事項」中的偏好和限制作為本次對話的持續規則。

**不做的事：**
- 不重新詢問摘要中已有的資訊。
- 不自行修改摘要中的結論（除非發現明顯錯誤，此時應先告知使用者）。
- 不跳過主動找檔步驟——必須先檢查檔案是否已上傳，再決定是否請使用者補傳。

## 四、工作階段節奏建議

### 一般法律工作

```
[對話 1] 案情了解 + 法律研究
         → 📋 產出接續摘要
[對話 2] 書狀撰寫（貼入摘要後開始）
         → 書狀完成
         → 📋 產出接續摘要（如需另一份書狀）
[對話 3] 書狀審查校對（貼入摘要 + 書狀檔案）
         → 定稿
```

### 大型案件（多份書狀）

```
[對話 1] 全案分析 → 📋 接續摘要
[對話 2] 書狀 A 撰寫 → 完成 → 📋 接續摘要
[對話 3] 書狀 B 撰寫 → 完成 → 📋 接續摘要
[對話 4] 書狀 C 撰寫 → 完成 → 📋 接續摘要
[對話 5] 整體審查（貼入所有書狀 + 摘要）
```

### Skill 開發

```
[對話 1] 需求討論 + Skill 設計 → 📋 接續摘要
[對話 2] Skill 撰寫（貼入摘要）→ 完成 → 📋 接續摘要
[對話 3] 測試 + 衝突檢測（貼入摘要）→ 定稿
```

## 五、快速指令

使用者可以用以下簡短指令觸發特定功能：

| 使用者說 | Claude 的動作 |
|---------|-------------|
| 「摘要一下」 | 執行確認清單 → 產出接續摘要（含啟動指令） |
| 「目前進度」 | 簡短回顧目前已完成和進行中的事項（不產出完整摘要） |
| 「工作地圖」 | 產出目前案件/專案的樹狀結構進度圖（不產出完整摘要） |
| 「開新對話要帶什麼」 | 產出接續摘要 + 強調 🔴 必要檔案清單 |
| 「這段可以忘掉」 | 記住使用者指定的段落不需帶入後續討論 |
| 「快速交接」 | 跳過確認清單，直接產出精簡版接續摘要（僅含下一步、必要檔案、啟動指令） |
| 「存檔」/「下載摘要」 | 將接續摘要以 `.md` 檔案輸出供下載保存 |

---

*在對的時間切換對話，比硬撐一個超長對話更能維持工作品質。*
